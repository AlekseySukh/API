Представлен обзор HTTP-методов с характеристиками, стандартными успешными статус-кодами и примерами их применения в контексте World Bank API (версия v2). Базовый URL: https://api.worldbank.org/v2/. Примечание: World Bank API в основном read-only (применяется GET). Остальные методы ниже описаны по стандарту HTTP и показаны как бы выглядели в API, которое поддерживает запись.

GET — получить ресурс: назначение — чтение данных (получить список/объект/страницу); безопасный (safe) — да (не должен менять состояние на сервере); идемпотентный — да; тело запроса — обычно не используется; кэширование — часто кэшируется. Успешные статус-коды для GET: 200 OK — данные успешно возвращены; 206 Partial Content — частичный ответ (обычно при Range, если поддерживается; для World Bank обычно нехарактерно); 304 Not Modified — при использовании кеширования (ETag/If-Modified-Since), если ресурс не менялся. 
Примеры (World Bank API) для GET: список стран https://api.worldbank.org/v2/country?format=json; информация по стране (Россия) https://api.worldbank.org/v2/country/RU?format=json; ВВП России (GDP current US$ — NY.GDP.MKTP.CD) https://api.worldbank.org/v2/country/RU/indicator/NY.GDP.MKTP.CD?format=json; тот же запрос с фильтром по датам https://api.worldbank.org/v2/country/RU/indicator/NY.GDP.MKTP.CD?date=2018:2022&format=json.

POST — создать ресурс / выполнить операцию: назначение — создать новый ресурс или выполнить операцию (command); безопасный — нет; идемпотентный — обычно нет; тело запроса — обычно да (JSON/формы); кэширование — обычно нет. Успешные статус-коды для POST: 201 Created — ресурс создан (часто с заголовком Location); 200 OK — операция выполнена, возвращён результат; 202 Accepted — запрос принят в обработку (асинхронно); 204 No Content — выполнено успешно, но без тела ответа. 
Пример (World Bank API не поддерживает создание ресурсов): POST https://api.worldbank.org/v2/some-resource с Content-Type: application/json и телом вида {"name":"example"}.

PUT — заменить ресурс целиком: назначение — полная замена ресурса по известному URL; безопасный — нет; идемпотентный — да; тело запроса — да (полное представление ресурса). Успешные статус-коды для PUT: 200 OK — ресурс обновлён и возвращён в ответе; 204 No Content — ресурс обновлён без тела ответа; 201 Created — если PUT создаёт ресурс (если его не было; зависит от API). Пример (форматно, “как бы выглядело”): PUT https://api.worldbank.org/v2/some-resource/123 с JSON вида {"id":123,"name":"new name","status":"active"}.

PATCH — частично изменить ресурс: назначение — частичное обновление (изменить некоторые поля); безопасный — нет; идемпотентный — зависит от реализации/формата patch (часто да, но не всегда); тело запроса — да (набор изменений). Успешные статус-коды для PATCH: 200 OK — ресурс обновлён и возвращён; 204 No Content — ресурс обновлён без тела ответа. 
Пример : PATCH https://api.worldbank.org/v2/some-resource/123 с JSON вида {"status":"archived"}.

DELETE — удалить ресурс: назначение — удаление ресурса; безопасный — нет; идемпотентный — да; тело запроса — обычно нет. Успешные статус-коды для DELETE: 204 No Content — успешно удалено, ответ без тела; 200 OK — успешно удалено, но сервер возвращает результат в теле; 202 Accepted — удаление принято и будет выполнено позже (асинхронно). 
Пример: DELETE https://api.worldbank.org/v2/some-resource/123.

HEAD — получить только заголовки (без тела): назначение — проверить наличие ресурса и метаданные (размер/кэш и т.п.); безопасный — да; идемпотентный — да; ответ — как на GET, но без тела. Успешные статус-коды для HEAD: 200 OK — ресурс существует; 304 Not Modified — ресурс не изменился (при кеш-проверке). 
Пример (теоретический): HEAD https://api.worldbank.org/v2/country/RU?format=json; не все публичные API корректно поддерживают HEAD, если не поддерживается — часто будет 405 Method Not Allowed.

OPTIONS — узнать доступные методы/настройки (в т.ч. CORS): назначение — узнать разрешённые методы/заголовки (часто используется как CORS preflight); безопасный — да; идемпотентный — да. Успешные статус-коды для OPTIONS: 200 OK — вернули список доступных методов/параметров; 204 No Content — без тела. 
Пример (теоретический): OPTIONS https://api.worldbank.org/v2/country?format=json.

Пагинация в World Bank API: World Bank API часто возвращает постраничные результаты; параметры page — номер страницы (начиная с 1) и per_page — число элементов на странице. В JSON-ответе первый элемент массива — метаданные (page/pages/per_page/total), второй элемент массива — данные. 
Примеры: задать размер страницы https://api.worldbank.org/v2/country?per_page=5&format=json; запросить конкретную страницу https://api.worldbank.org/v2/country?per_page=5&page=2&format=json. Как “выгрузить всё” корректно: запросить page=1, прочитать pages из метаданных, затем пройти циклом по page=1..pages, собирая данные; пример стартового запроса https://api.worldbank.org/v2/country?per_page=100&format=json. Нюанс: per_page не бесконечный; в публичных API часто есть лимит на per_page, если поставить слишком большое значение, сервер может ограничить до максимума, или вернуть ошибку, или отдать меньше без явного сообщения, поэтому лучше ориентироваться на pages в метаданных, а не рассчитывать на один “огромный” запрос.

Короткое резюме по применимости к World Bank API: практически все рабочие запросы — это GET; успешный статус для чтения — обычно 200 OK; пагинация — page + per_page, метаданные в первом элементе JSON-массива.
